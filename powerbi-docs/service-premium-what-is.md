# <a name="what-is-power-bi-premium"></a>Power BI Premium คืออะไร?

Power BI Premium มอบทรัพยากรเฉพาะ และปรับปรุงเมื่อต้องการเรียกใช้บริการ Power BI สำหรับองค์กรของคุณ ตัวอย่างเช่น:

- ปริมาณและประสิทธิภาพการทำงาน
- ความยืดหยุ่นในการสิทธิ์การใช้งานโดยความจุ
- รวมด้วยตนเองและองค์กร BI
- ขยาย BI ภายในองค์กร ด้วย Power BI Report Server
- การสนับสนุนสำหรับข้อมูลประชากรประเทศตามภูมิภาค (หลายทางภูมิศาสตร์)
- แชร์ข้อมูลกับผู้อื่นโดยไม่ต้องซื้อสิทธิ์การใช้งานต่อผู้ใช้

บทความนี้มีไว้เพื่อให้รายละเอียดเชิงลึกเกี่ยวกับคุณลักษณะของ Power BI Premium - ทุกอันที่จริงไม่ เพียงสัมผัสพื้นผิว กรณีจำเป็น ให้ลิงก์ไปยังบทความเพิ่มเติมที่มีรายละเอียดเพิ่มเติม

## <a name="subscriptions-and-licensing"></a>สมัครใช้งานและสิทธิ์การใช้งาน

Power BI Premium คือการสมัครใช้งาน Office 365 ระดับผู้เช่าที่มีอยู่ในสองกลุ่ม SKU (Stock-Keeping Unit) ได้แก่:

- **EM** Sku (EM1-EM3) สำหรับการฝัง จำเป็นต้องมีข้อผูกมัดรายปี เรียกเก็บเงินรายเดือน
- **P** Sku (P1 P3) สำหรับคุณลักษณะที่ฝังและองค์กร จำเป็นต้องเป็นข้อตกลงแบบรายเดือน หรือรายปี เรียกเก็บเงินรายเดือน และมีสิทธิ์การใช้งานการติดตั้งเซิร์ฟเวอร์รายงาน Power BI ภายในองค์กร

วิธีอื่นคือการ ซื้อการ**Azure Power BI Embedded**สมัครใช้งาน ซึ่งมีเดียว**A**ครอบครัว SKU (A1 A6) สำหรับฝังตัวและการทดสอบความจุเท่านั้น Sku ทั้งหมดส่งวี-คอร์เพื่อสร้างความจุ แต่ EM Sku ถูกจำกัดสำหรับการฝังมาตราส่วนขนาดเล็ก EM1, EM2, A1 และ A2 Sku มีน้อยกว่าสี่วี-คอร์ไม่ทำงานบนโครงสร้างพื้นฐานเฉพาะ

ในขณะที่โฟกัสของบทความนี้อยู่บน P Sku มากของสิ่งที่อธิบายไว้จะเกี่ยวข้องกับ Sku A Azure SKU ไม่มีสัญญาผูกมัดด้านเวลาและมีการเรียกเก็บเงินเป็นรายชั่วโมง ซึ่งตรงกันข้ามกับ SKUs ของการสมัครใช้งานระดับ Premium ซึ่งมีความยืดหยุ่นเต็มที่โดยช่วยให้สามารถขยายขนาด ลดขนาด หยุดชั่วคราว ดำเนินต่อ และลบได้ 

Azure Power BI Embedded อยู่เป็นส่วนใหญ่นอกขอบเขตสำหรับบทความนี้ แต่จะถูกอธิบายไว้ในการ[ทดสอบวิธี](service-premium-capacity-optimize.md#testing-approaches)ส่วนของบทความจุ Premium ที่ปรับให้เหมาะสมเป็นตัวเลือกทางปฏิบัติ และเศรษฐกิจเพื่อทดสอบ และวัดปริมาณงาน เมื่อต้องการเรียนรู้เพิ่มเติมเกี่ยวกับ Sku Azure ดู[เอกสารประกอบของ Azure Power BI Embedded](https://azure.microsoft.com/services/power-bi-embedded/)

### <a name="purchasing"></a>การจัดซื้อ

ผู้ดูแลระบบในศูนย์การจัดการ Microsoft 365 จะซื้อการสมัครใช้งาน Power BI Premium โดยเฉพาะ เฉพาะผู้ดูแลระบบส่วนกลางของ Office 365 หรือผู้ดูแลระบบการเรียกเก็บเงินสามารถซื้อ Sku เมื่อซื้อ ผู้เช่าที่ได้รับตัวเลขที่สอดคล้องกันของวี-คอร์เพื่อกำหนดให้กับกำลังการผลิต เรียกว่า*วี-คอร์ร่วม* ตัวอย่างเช่น การซื้อ P3 SKU มีผู้เช่าที่มี 32 วี-คอร์ เมื่อต้องการเรียนรู้เพิ่มเติม ดู[วิธีการซื้อ Power BI Premium](service-admin-premium-purchase.md)

## <a name="dedicated-capacities"></a>ความจุเฉพาะ

ด้วย Power BI Premium คุณได้รับ*มุ่งเน้นความจุ* ข้ามความจุที่ใช้ร่วมกันที่ปริมาณงานที่เรียกใช้บนแหล่งทรัพยากรการคำนวณที่ใช้ร่วมกันกับลูกค้าคนอื่น ๆ ความจุเฉพาะถูกใช้เฉพาะโดยองค์กร จะมีการแยกออกมาต่างหากกับแหล่งข้อมูลคำนวณเฉพาะที่มีประสิทธิภาพขึ้น และสอดคล้องกันสำหรับเนื้อหาที่โฮสต์ 

พื้นที่ทำงานอยู่ภายในกำลังการผลิต ผู้ใช้ Power BI แต่ละคนมีพื้นที่ทำงานส่วนบุคคลเรียกว่า**พื้นที่ทำงานของฉัน** สามารถสร้างพื้นที่ทำงานเพิ่มเติมเพื่อช่วยในการทำงานร่วมกันและการปรับใช้ และพื้นที่เหล่านี้เรียกว่า**พื้นที่ทำงานของแอป** ตามค่าเริ่มต้น พื้นที่ทำงาน รวมถึงพื้นที่ทำงานส่วนบุคคล จะถูกสร้างขึ้นในความจุที่ใช้ร่วมกัน เมื่อคุณมีความจุ Premium พื้นที่ทำงานทั้งพื้นที่ทำงานของฉันและแอสามารถกำหนดให้กับกำลังการผลิตพรีเมียม

### <a name="capacity-nodes"></a>โหนดกำลังการผลิต

ตามที่อธิบายไว้ในการ[สมัครใช้งานและสิทธิ์การใช้งาน](#subscriptions-and-licensing)ส่วน มีตระกูล Power BI Premium SKU ที่สอง: **EM**และ**P** ทั้งหมด Power BI Premium Sku จะพร้อมใช้งานเป็นความจุ*โหน*แต่ละบรรทัดแทนจำนวนชุดของแหล่งข้อมูลที่ประกอบด้วยตัวประมวลผล หน่วยความจำ และที่เก็บข้อมูล นอกเหนือจากแหล่งข้อมูล แต่ละ SKU มีขีดจำกัดในการดำเนินงานจำนวนการเชื่อมต่อ DirectQuery และเชื่อมต่อแบบ Live ต่อวินาที และรีเฟรชจำนวนแบบขนาน

ตัวประมวลผลจะทำตามจำนวนชุดของวี-คอร์ ซึ่งแบ่งออกเท่า ๆ กันระหว่าง Frontend และ Backend

**วี-คอร์ Backend** รับผิดชอบสำหรับฟังก์ชันการทำงานของ Power BI รวมถึงการประมวลผลคิวรี การจัดการแคช การเรียกใช้การบริการ R การรีเฟรชแบบจำลอง การประมวลผลภาษาธรรมชาติ (ถามตอบ) และการแสดงผลฝั่งเซิร์ฟเวอร์ในรูปแบบรายงานและรูป วี-คอร์ Backend จะถูกกำหนดให้เป็นจำนวนคงของหน่วยความจำที่ใช้เป็นหลักเพื่อโฮสต์แบบจำลอง ชุดข้อมูลที่เรียกว่าใช้งานอยู่

**วี-คอร์ Frontend**รับผิดชอบสำหรับ web service แดชบอร์ด และรายงานการบริหารเอกสาร จัดการสิทธิ์การเข้าถึง จัดกำหนดการ APIs อัปโหลด และดาวน์โหลด และโดยทั่วไปสำหรับทุกอย่างที่เกี่ยวข้องกับผู้ใช้ประสบการณ์การใช้งาน

เก็บข้อมูลถูกตั้งค่าเป็น**100 TB ต่อโหนดความจุ**

แหล่งข้อมูลและขีดจำกัดของแต่ละ SKU Premium (และ equivalently ขนาด A SKU) อธิบายไว้ในตารางต่อไปนี้:

### <a name="capacity-workloads"></a>ปริมาณงานกำลังการผลิต

ปริมาณงานสำหรับความจุเป็นบริการที่กำหนดไว้สำหรับผู้ใช้ ตามค่าเริ่มต้น Premium ความจุ Azure สนับสนุนเฉพาะปริมาณชุดข้อมูลที่เกี่ยวข้องกับการเรียกใช้คิวรี Power BI ปริมาณงานชุดข้อมูลไม่สามารถถูกปิดใช้งาน ปริมาณงานเพิ่มเติมสามารถเปิดใช้งานสำหรับ[AI (Cognitive Services)](https://powerbi.microsoft.com/blog/easy-access-to-ai-in-power-bi-preview/), [Dataflows](service-dataflows-overview.md#dataflow-capabilities-on-power-bi-premium)และ[แบ่งรายงาน](paginated-reports-save-to-power-bi-service.md)ได้ ปริมาณงานเหล่านี้ได้รับการสนับสนุนในการสมัครใช้งาน Premium เท่านั้น 

แต่ละปริมาณงานเพิ่มเติมให้กำหนดค่าสูงสุดหน่วยความจำ (เป็นเปอร์เซ็นต์ของผลรวมหน่วยความจำที่พร้อมใช้งาน) ที่ใช้ได้ โดยที่ปริมาณงาน ค่าเริ่มต้นสำหรับหน่วยความจำสูงสุดจะถูกกำหนด โดย SKU คุณสามารถขยายทรัพยากรพร้อมใช้งานความจุของคุณ โดยการเปิดใช้งานเท่านั้นปริมาณงานเหล่านั้นเพิ่มเติมเมื่อพวกเขาคุ้นเคย และคุณสามารถเปลี่ยนการตั้งค่าเฉพาะเมื่อคุณได้กำหนดตั้งค่าเริ่มต้นจะประชุมของคุณต้องการทรัพยากรความจุหน่วยความจำ ปริมาณงานคุณสามารถเปิดใช้งาน และกำหนดค่าสำหรับผู้ดูแลความจุโดยความจุที่ โดยใช้**ตั้งค่าความจุ**ในการ[พอร์ทัลผู้ดูแลระบบ](service-admin-portal.md)หรือ โดยใช้การ[ความจุ REST APIs](https://docs.microsoft.com/rest/api/power-bi/capacities)ได้  

![เปิดใช้งานปริมาณงาน](media/service-admin-premium-workloads/admin-portal-workloads.png)

### <a name="how-capacities-function"></a>วิธีฟังก์ชันกำลังการผลิต

เวลา บริการของ Power BI ทำให้การใช้ที่ดีที่สุดของทรัพยากรความจุขณะไม่เกินขีดจำกัดที่กำหนดบนกำลังการผลิต

การดำเนินการกำลังการผลิตจะถูกจัดประเภทเป็น*แบบโต้ตอบ*หรือ*พื้นหลัง* การทำงานแบบโต้ตอบประกอบด้วยการแสดงคำขอ และการตอบสนองโต้ตอบกับผู้ใช้ (การกรอง การคิวรีถามตอบ และอื่น ๆ) โดยทั่วไปแล้ว นำเข้าการสอบถามแบบจำลองคือ หน่วยความจำที่ใช้ทรัพยากร ขณะสอบถามแบบจำลอง DirectQuery และเชื่อมต่อสด CPU สูง การทำงานแบบเบื้องหลังประกอบด้วยกระแสข้อมูลและการรีเฟรชแบบจำลองการนำเข้า และการแคชคิวรีของแดชบอร์ด

จะต้องทำความเข้าใจว่า การดำเนินการแบบโต้ตอบจะเสมอจัดลำดับความสำคัญเหนือพื้นหลังการดำเนินการเพื่อให้แน่ใจว่าประสบการณ์การใช้งานของผู้ใช้ที่เป็นไปได้ดีที่สุด ถ้ามีทรัพยากรไม่เพียงพอ การทำงานแบบเบื้องหลังจะถูกเพิ่มไปยังคิวสำหรับการประมวลผลเมื่อเพิ่มแหล่งข้อมูล การดำเนินการพื้นหลัง เช่นรีเฟรชชุดข้อมูล จะหยุดกลาง โดยบริการ Power BI และเพิ่มลงในคิว

นำเข้าแบบจำลองต้องโหลดลงในหน่วยความจำเพื่อให้พวกเขาสามารถสอบถาม หรือรีเฟรช บริการ Power BI จัดการหน่วยความจำที่ใช้งาน โดยใช้อัลกอริทึมเพื่อให้แน่ใจว่าใช้สูงสุดของหน่วยความจำที่พร้อมใช้งานที่ซับซ้อน และอาจก่อให้เกิดความจุที่ยอมรับมากเกินไป: ใน ขณะที่อาจเป็นไปได้สำหรับความจุเก็บนำเข้าหลายแบบจำลอง (สูงสุด 100 TB ต่อความจุ Premium) เมื่อการเก็บข้อมูลดิสก์รวมเกินหน่วยความจำที่ได้รับการสนับสนุน (และหน่วยความจำเพิ่มเติมจำเป็นสำหรับการสอบถามและรีเฟรช), แล้วพวกเขาไม่ทั้งหมดสามารถโหลดลงในหน่วยความจำที่ ในครั้งเดียวกัน

นำเข้าแบบจำลองจะโหลดลงในดังนั้น และเอาออกจากหน่วยความจำตามการใช้งาน แบบจำลองนำเข้าจะโหลดเมื่อสอบถาม (แบบโต้ตอบการดำเนินการ) และยังไม่ได้ ในหน่วยความจำ หรือ เมื่อการรีเฟรช (พื้นหลังการดำเนินการ)

การลบแบบจำลองจากหน่วยความจำจะเรียกว่า*eviction* การดำเนินการ Power BI สามารถทำได้อย่างรวดเร็วโดยขึ้นอยู่กับขนาดของแบบจำลองได้ หากความจุนั้นไม่พบความต้องการใช้งานหน่วยความจำใด ๆ แบบจำลองถูกโหลดลงในหน่วยความจำและยังคงอยู่ที่นั่น อย่างไรก็ตาม เมื่อหน่วยความจำไม่เพียงพอจะพร้อมใช้งานการโหลดแบบ บริการของ Power BI จะต้องขึ้นฟรีหน่วยความจำ ว่างหน่วยความจำ โดยตรวจสอบแบบจำลองที่มีใช้งาน โดยการค้นหาแบบจำลองซึ่งยังไม่ได้ใช้ในไม่กี่นาทีสามล่าสุด\[ [1](#endnote-1)\]และลดเหล่านั้น ถ้าไม่มีแบบจำลองที่ไม่ได้ใช้งานที่จะลดสัดส่วน บริการของ Power BI จะค้นหาเพื่อลดสัดส่วนแบบจำลองที่โหลดสำหรับการทำงานแบบเบื้องหลัง วิธีสุดท้าย หลังจาก 30 วินาทีของความพยายามที่ล้มเหลว\[ [1](#endnote-1)\]จะล้มเหลวการดำเนินการแบบโต้ตอบ ในกรณีนี้ ผู้ใช้รายงานได้รับแจ้งเกี่ยวกับความล้มเหลวด้วยคำแนะนำให้ลองอีกครั้งในอีกสักครู่ ในบางกรณี แบบจำลองอาจยกเลิกการโหลดจากหน่วยความจำเนื่องจากการดำเนินการบริการ

สิ่งสำคัญคือต้อง stress กำจัดชุดข้อมูลว่าปกติ และคาดหวังลักษณะการทำงาน ซึ่งเป็นความพยายามที่จะเพิ่มการใช้งานหน่วยความจำให้สูงสุดโดยการโหลดและการยกเลิกการโหลดแบบจำลองที่มีขนาดรวมกันเกินกว่าหน่วยความจำที่มีอยู่ โดยเป็นไปตามการออกแบบและเปิดเผยอย่างตรงไปตรงมาต่อผู้ใช้รายงาน อัตราการลดสัดส่วนข้อมูลสูงไม่ได้หมายความว่าความจุมีปริมาณไม่เพียงพอ อย่างไรก็ตามอาจกลายเป็นข้อจำกัดได้หากการตอบสนองของการคิวรีและการรีเฟรชจะได้รับผลกระทบเนื่องจากอัตราการลดสัดส่วนข้อมูลสูง

รีเฟรชของแบบจำลองการนำเข้าได้เสมอหน่วยความจำจำนวนมาก ตามแบบจำลองต้องโหลดลงในหน่วยความจำ หน่วยความจำเพิ่มเติมที่ถูกต้องสำหรับการประมวลผล การรีเฟรชเต็มรูปแบบอาจใช้จำนวนหน่วยความจำเป็นสองเท่าโดยประมาณตามที่แบบจำลองกำหนด ซึ่งทำให้แน่ใจแบบจำลองสามารถสอบถามได้แม้ในขณะที่กำลังประมวลผล เนื่องจากแบบสอบถามจะถูกส่งไปยังรูปแบบที่มีอยู่ จนกว่าการรีเฟรชเสร็จสมบูรณ์ และแบบจำลองข้อมูลใหม่จะพร้อมใช้งาน รีเฟรชแบบเพิ่มหน่วยจะจำเป็นต้องมีหน่วยความจำน้อยกว่า และอาจทำให้เสร็จสมบูรณ์ได้อย่างรวดเร็ว และดังนั้น สามารถวางจำหน่ายจริงลดความดันบนแหล่งทรัพยากรความจุ นอกจากนี้ การรีเฟรชยังอาจใช้ CPU สูงสำหรับแบบจำลอง โดยเฉพาะการรีเฟรชที่มีการเปลี่ยนแปลงของ Power Query ซับซ้อน หรือตาราง/คอลัมน์จากการคำนวณที่ซับซ้อนหรือยึดตามตารางขนาดใหญ่ได้

รีเฟรช เช่นคิวรี จำเป็นต้องมีแบบจำลองถูกโหลดลงในหน่วยความจำ ถ้ามีหน่วยความจำไม่เพียงพอ บริการของ Power BI จะพยายามทำการลดสัดส่วนแบบจำลองที่ไม่ได้ใช้งาน และถ้าไม่เป็นไปได้ (แบบจำลองทั้งหมดมีการใช้งานอยู่) งานการรีเฟรชถูกจัดคิวไว้ การรีเฟรชโดยทั่วไปแล้ว CPU สูง แม้มากกว่าการสอบถาม ด้วยเหตุนี้จึงมีข้อจำกัดด้านความจุของจำนวนการรีเฟรชพร้อมกัน ซึ่งตั้งค่าเป็น 1.5 x จำนวนวี-คอร์ Backend โดยการปัดเศษขึ้น ถ้ามีการรีเฟรชพร้อมกันมากเกินไป การรีเฟรชตามกำหนดเวลาจะถูกจัดคิวไว้ เมื่อสถานการณ์เหล่านี้เกิดขึ้น การรีเฟรชจะใช้เวลานานจึงจะเสร็จสมบูรณ์ รีเฟรชตามความต้องการเช่นทริกเกอร์ โดยคำขอของผู้ใช้ หรือการเรียกใช้ API จะลองอีกครั้งสามครั้ง\[ [1](#endnote-1)\] ถ้ายังคงไม่เพียงพอทรัพยากร การรีเฟรชจะล้มเหลว

### <a name="regional-support"></a>สนับสนุนภูมิภาค

เมื่อสร้างความจุใหม่ ผู้ดูแลระบบส่วนกลางของ Office 365 และผู้ดูแลระบบบริการ Power BI สามารถระบุ ภูมิภาคที่กำหนดพื้นที่ทำงานความจุจะอยู่ ซึ่งเรียกว่า**หลายทางภูมิศาสตร์** มีหลายทางภูมิศาสตร์ องค์กรสามารถตามข้อมูลประชากรประเทศข้อกำหนด โดยการปรับใช้เนื้อหาไปยังศูนย์ข้อมูลในภูมิภาคเฉพาะ แม้ว่าจะเป็นแตกต่างจากภูมิภาคการสมัครใช้งาน Office 365 อยู่ เมื่อต้องการเรียนรู้เพิ่มเติม ดู[สนับสนุนหลายทางภูมิศาสตร์สำหรับ Power BI Premium](service-admin-premium-multi-geo.md)

### <a name="capacity-management"></a>จัดการกำลังการผลิต

จัดการความจุ Premium เกี่ยวข้องกับการสร้าง หรือลบความจุ กำหนดผู้ดูแลระบบ กำหนดพื้นที่ทำงาน กำหนดปริมาณงาน และการตรวจสอบการปรับปรุงประสิทธิภาพการทำงานความจุที่ปรับให้เหมาะสม 

Office 365 ผู้ดูแลระบบส่วนกลางและผู้ดูแลระบบบริการ Power BI สามารถสร้างความจุ Premium จากแกน v ที่พร้อมใช้งาน หรือปรับเปลี่ยนความจุ Premium ที่มีอยู่ เมื่อมีสร้างความจุ ระบุขนาดความจุและภูมิภาคทางภูมิศาสตร์ และผู้ดูแลความจุอย่างน้อยหนึ่งถูกกำหนดให้ 

เมื่อมีสร้างความจุ งานดูแลระบบส่วนใหญ่เสร็จสมบูรณ์ในการ[พอร์ทัลผู้ดูแลระบบ](service-admin-portal.md)

![พอร์ทัลผู้ดูแลระบบ](media/service-premium-what-is/premium-admin-portal.png)

ผู้ดูแลความจุสามารถกำหนดพื้นที่ทำงานความจุ จัดการสิทธิ์ของผู้ใช้ และกำหนดผู้ดูแลระบบอื่น ๆ ผู้ดูแลความจุยังสามารถกำหนดปริมาณงาน ปรับปันหน่วยความจำ และถ้าจำเป็น รีสตาร์ทความจุ ตั้งค่าการดำเนินการในกรณีที่โอเวอร์โหลดกำลังการผลิต

![พอร์ทัลผู้ดูแลระบบ](media/service-premium-what-is/premium-admin-portal-mgmt.png)

ผู้ดูแลความจุยังสามารถตรวจสอบว่า ความจุที่ใช้งานได้อย่างราบรื่น พวกเขาสามารถตรวจสอบขวาสถานภาพความจุ ในพอร์ทัลผู้ดูแล หรือ โดยใช้แอปเมตริกของความจุ Premium

เมื่อต้องการเรียนรู้เพิ่มเติมเกี่ยวกับการสร้างความจุ กำหนดผู้ดูแลระบบ และกำหนดพื้นที่ทำงาน ดู[ความจุ Premium จัดการ](service-premium-capacity-manage.md) เมื่อต้องการเรียนรู้เพิ่มเติมเกี่ยวกับบทบาท ดู[บทบาทผู้ดูแลระบบที่เกี่ยวข้องกับ Power BI](service-admin-administering-power-bi-in-your-organization.md#administrator-roles-related-to-power-bi)

### <a name="monitoring"></a>การตรวจสอบ

ตรวจสอบความจุ Premium มีผู้ดูแลระบบที่ มีความเข้าใจเกี่ยวกับวิธีการดำเนินการกำลังการผลิต กำลังการผลิตสามารถตรวจสอบ โดยใช้พอร์ทัลผู้ดูแลและ[แอป Power BI Premium ความจุเมตริก](https://app.powerbi.com/groups/me/getapps/services/capacitymetrics)ได้

ตรวจสอบในพอร์ทัลมีมุมมองอย่างรวดเร็ว ด้วยเมตริกระดับสูงที่ระบุว่า การโหลดที่วางและทรัพยากรที่ใช้ได้ โดยความจุของคุณ ค่าเฉลี่ย เหนือเจ็ดวันที่ผ่านมา 

![พอร์ทัลผู้ดูแลระบบ](media/service-premium-what-is/premium-admin-portal-health.png)

การ**Power BI Premium ความจุเมตริก**แอปมีข้อมูลเชิงลึกมากที่สุดเป็นวิธีการดำเนินการกำลังการผลิตของคุณ แอปที่มีแดชบอร์ดระดับสูง และรายงานโดยละเอียดเพิ่มเติม

![แดชบอร์ดแอปวัด](media/service-admin-premium-monitor-capacity/app-dashboard.png)

จากแดชบอร์ดของแอป คุณสามารถคลิกเซลล์เมตริกเมื่อต้องเปิดรายงานที่มีข้อมูลเชิงลึก รายงานมีเมตริกเชิงลึกและความสามารถในการกรองเพื่อเจาะลึกบนข้อมูลที่สำคัญที่สุดคุณจำเป็นต้องเก็บความจุของคุณทำงานได้อย่างราบรื่น

![จำนวนเวลารอคิวรีที่ถึงระดับสูงสุดเป็นระยะ ๆ แสดงถึงความอิ่มตัวของ CPU ที่อาจเกิดขึ้น](media/service-premium-capacity-scenarios/peak-query-wait-times.png)

เมื่อต้องการเรียนรู้เพิ่มเติมเกี่ยวกับการตรวจสอบความจุ ดู[ตรวจสอบในพอร์ทัลผู้ดูแลระบบ Power b](service-admin-premium-monitor-portal.md)และ[ตรวจสอบ ด้วยแอป Power BI Premium ความจุเมตริก](service-admin-premium-monitor-capacity.md)

### <a name="optimizing-capacities"></a>ความจุที่ปรับให้เหมาะสม

ทำให้การใช้ที่ดีที่สุดของความจุของคุณเป็นสิ่งสำคัญในการ assuring รับผู้ใช้ประสิทธิภาพการทำงาน และคุณได้รับประโยชน์สูงสุดสำหรับการลงทุน Premium โดยการตรวจสอบเมตริกหลัก ผู้ดูแลระบบสามารถกำหนดวิธีที่ดีที่สุดเพื่อแก้ไขปัญหาคอขวด และดำเนินการที่จำเป็นหรือไม่ เมื่อต้องการเรียนรู้เพิ่มเติม ดู[ความจุ Premium ที่ปรับให้เหมาะสม](service-premium-capacity-optimize.md)และ[สถานการณ์ความจุ Premium](service-premium-capacity-scenarios.md)

### <a name="capacities-rest-apis"></a>ความจุ REST APIs

คอลเลกชันของรวม Power BI REST APIs [Api ที่ความจุ](https://docs.microsoft.com/rest/api/power-bi/capacities) กับ Api ผู้ดูแลระบบสามารถเขียนโปรแกรมจัดการลักษณะต่าง ๆ ของความจุ Premium ของคุณ รวมถึงการเปิดใช้งาน และปิดใช้งานปริมาณงาน กำหนดพื้นที่ทำงานกับความจุ และอื่น ๆ

## <a name="large-datasets"></a>ชุดข้อมูลขนาดใหญ่

ขึ้นอยู่กับ SKU, Power BI Premium รองรับไฟล์แบบจำลอง Power BI Desktop (.pbix) อัปโหลดสูงสุด**10 GB**ขนาด เมื่อโหลด แบบจำลองสามารถแล้วเผยแพร่ไปยังพื้นที่ทำงานการกำหนดความจุ Premium ชุดข้อมูลนั้นถูกรีเฟรชเพื่อสูง**12 GB**ขนาดได้

### <a name="size-considerations"></a>ข้อควรพิจารณาขนาด

รูปแบบขนาดใหญ่อาจใช้ทรัพยากร คุณควรมีอย่างน้อย P1 SKU สำหรับแบบจำลองที่มีขนาดใหญ่กว่า 1 GB แม้ว่าการประกาศรูปแบบขนาดใหญ่ไปยังพื้นที่ทำงานการสำรองข้อมูล โดย A Sku จนถึง A3 อาจทำงาน การรีเฟรชเหล่านั้นจะไม่

ตารางต่อไปนี้อธิบายถึง SKU ที่แนะนำสำหรับขนาด.pbix ต่าง ๆ

   |SKU  |ขนาดของ.pbix   |
   |---------|---------|
   |P1    | < 3 GB        |
   |P2    | < 6 GB        |
   |P3, P4, P5    | สูงสุด 10 GB   |

Power BI Embedded A4 SKU มีค่าเท่ากับ P1 SKU, A5 = P2 และ A6 = P3 โปรดทราบว่าการเผยแพร่แบบจำลองขนาดใหญ่ไปยัง A SKU และ EM SKU อาจส่งคืนข้อผิดพลาดที่ไม่ได้ระบุถึงข้อผิดพลาดด้านข้อจำกัดของขนาดแบบจำลองในความจุที่ใช้ร่วมกัน ข้อผิดพลาดในการรีเฟรชของแบบจำลองขนาดใหญ่ใน A SKU และ EM SKU อาจเน้นไปที่ช่วงหมดเวลา 

ไฟล์.pbix ของคุณแสดงข้อมูลในการ*สูงบีบอัดสถานะ* ข้อมูลจะมีแนวโน้มว่าจะขยายหลาย ๆ ครั้งเมื่อโหลดในหน่วยความจำ และจากที่นั่น อาจจะขยายหลายครั้งในระหว่างการรีเฟรชข้อมูล

รีเฟรชตามกำหนดการของชุดข้อมูลขนาดใหญ่สามารถใช้เวลานาน และสามารถใช้ทรัพยากร สิ่งสำคัญคือต้องจัดกำหนดการการรีเฟรชที่ซ้อนกันมากเกินไปไม่ แนะนำ[รีเฟรชแบบเพิ่มหน่วย](service-premium-incremental-refresh.md)ถูกกำหนดค่า เนื่องจากเป็นเร็ว เชื่อถือได้มากกว่า และใช้ทรัพยากรน้อยลง

การโหลดรายงานเริ่มต้นของชุดข้อมูลขนาดใหญ่สามารถใช้เวลานานถ้าถูกสักตั้งแต่ครั้งสุดท้ายที่ใช้ชุดข้อมูล แถบโหลดสำหรับรายงานยาวการโหลดแสดงความคืบหน้าของการโหลด

ในขณะที่หน่วยความจำต่อคิวรีและข้อจำกัดเวลาสูงกว่าในความจุ Premium แนะนำให้คุณใช้ตัวกรองและตัวแบ่งส่วนข้อมูลเพื่อจำกัดเท่านั้น ที่จำเป็น

## <a name="incremental-refresh"></a>การรีเฟรชแบบเพิ่ม

รีเฟรชแบบเพิ่มหน่วยให้ส่วนสำคัญของการมี และรักษาชุดข้อมูลขนาดใหญ่ใน Power BI Premium รีเฟรชแบบเพิ่มหน่วยมีประโยชน์มากมาย ตัวอย่าง เวลาที่รีเฟรชจะเร็วขึ้นเนื่องจากเฉพาะข้อมูลที่มีความต้องการเปลี่ยนแปลงการรีเฟรช การรีเฟรชเชื่อถือได้มากขึ้นเนื่องจากไม่จำเป็นเพื่อรักษานานเชื่อมต่อกับแหล่งข้อมูลที่เปลี่ยนแปลงได้ ทรัพยากรลดการใช้งานเนื่องจากข้อมูลน้อยลงเพื่อรีเฟรชช่วยลดปริมาณการใช้โดยรวมของหน่วยความจำและทรัพยากรอื่น ๆ กำหนดนโยบายการรีเฟรชแบบเพิ่มหน่วยใน**Power BI Desktop**และนำไปใช้เมื่อเผยแพร่ไปยังพื้นที่ทำงานในความจุ Premium 

![รายละเอียดการรีเฟรช](media/service-premium-incremental-refresh/refresh-details.png)

เมื่อต้องการเรียนรู้เพิ่มเติม ดู[รีเฟรชแบบเพิ่มหน่วยใน Power BI Premium](service-premium-incremental-refresh.md)

## <a name="paginated-reports"></a>รายงานที่มีการแบ่งหน้า

รายงานแบบแบ่งหน้า ได้รับการสนับสนุนบน P1 P3 และ Sku A4_A6 จะอิงตามเทคโนโลยี Report Definition Language (RDL) ใน SQL Server Reporting Services ในขณะที่ยึดตามเทคโนโลยี RDL ไม่เหมือนกับเซิร์ฟเวอร์รายงาน Power BI ซึ่งเป็นแพลตฟอร์รายงานสามารถดาวน์โหลดได้ที่คุณสามารถติดตั้งภายในองค์กร ยัง มาพร้อมกับ Power BI Premium รายงานแบบแบ่งหน้าถูกจัดรูปแบบให้พอดีกับหน้าที่สามารถพิมพ์ หรือใช้ร่วมกัน คุณสามารถแสดงข้อมูลในตาราง แม้ว่าตารางครอบคลุมหลายหน้า โดยใช้ฟรี[ **ตัวสร้างรายงาน BI Power** ](https://go.microsoft.com/fwlink/?linkid=2086513)แอปพลิเคชัน Windows Desktop ผู้เขียนผู้ใช้ที่มีการแบ่งหน้ารายงาน และเผยแพร่ไปยังบริการ

ใน Power BI Premium, Paginated รายงานมีปริมาณงานที่ต้องเปิดใช้งานสำหรับความจุ โดยใช้พอร์ทัลผู้ดูแล ผู้ดูแลความจุสามารถเปิดใช้งาน และจากนั้น ระบุจำนวนหน่วยความจำเป็นเปอร์เซ็นต์ของความจุทรัพยากรหน่วยความจำโดยรวม ต่างจากชนิดอื่น ๆ ของปริมาณงาน Premium เรียกใช้รายงานแบบแบ่งหน้ามีช่องว่างที่มีอยู่ภายในความจุ ใช้หน่วยความจำสูงสุดที่ระบุสำหรับพื้นที่นี้ไม่ว่าปริมาณงานที่ทำงานอยู่หรือไม่ ค่าเริ่มต้นคือ 20% 

เมื่อต้องการเรียนรู้เพิ่มเติม ดู[แบ่งรายงานใน Power BI Premium](paginated-reports-report-builder-power-bi.md) เมื่อต้องการเรียนรู้เพิ่มเติมเกี่ยวกับการเปิดใช้งานปริมาณงานรายงาน Paginated ดู[ปริมาณงานการกำหนดค่า](service-admin-premium-workloads.md)

## <a name="power-bi-report-server"></a>เซิร์ฟเวอร์รายงาน Power BI
 
มาพร้อมกับ ด้วย Power BI Premium เซิร์ฟเวอร์รายงาน Power BI จะมี*ภายในองค์กร*เซิร์ฟเวอร์รายงานกับพอร์ทัลเว็บ คุณสามารถสร้างสภาพแวดล้อมภายในองค์กรของคุณ BI และแจกจ่ายรายงานหลังไฟร์วอลล์ขององค์กรของคุณ เซิร์ฟเวอร์รายงานให้ผู้ใช้เข้าถึงที่สมบูรณ์ แบบโต้ตอบ และความสามารถสำหรับองค์กรรายงานของ SQL Server Reporting Services ผู้ใช้สามารถสำรวจข้อมูลภาพ และค้นหารูปแบบเพื่อทำการตัดสินใจได้เร็วขึ้นดี อย่างรวดเร็ว เซิร์ฟเวอร์รายงานมีการกำกับดูแลตามข้อกำหนดของคุณเอง ถ้าเวลามา และเมื่อเซิร์ฟเวอร์รายงาน Power BI ทำให้ง่ายต่อการโยกย้ายไปยัง cloud ที่องค์กรของคุณสามารถใช้ประโยชน์จากฟังก์ชันการทำงานทั้งหมดของ Power BI Premium ทั้งหมด

เมื่อต้องการเรียนรู้เพิ่มเติม ดู[เซิร์ฟเวอร์รายงาน Power BI](report-server/get-started.md)

## <a name="unlimited-content-sharing"></a>แชร์เนื้อหาที่ไม่จำกัด

ด้วย Premium ทุกคน ไม่ว่าพวกเขาอยู่ภายใน หรือภาย นอกองค์กรของคุณสามารถดูเนื้อหา Power BI ของคุณรวมถึงรายงานแบบโต้ตอบ และมีการแบ่งหน้าโดยไม่ต้องซื้อสิทธิ์การใช้งานแต่ละรายการ 

![แชร์เนื้อหา](media/service-premium-what-is/premium-sharing.png)

Premium ช่วยให้แพรเนื้อหาโดยผู้ใช้ Pro โดยไม่จำเป็นต้องมีสิทธิ์การใช้งาน Pro สำหรับผู้รับที่ดูเนื้อหา สิทธิ์การใช้งาน pro จะจำเป็นสำหรับผู้สร้างเนื้อหา ผู้สร้างการเชื่อมต่อกับแหล่งข้อมูล แบบจำลองข้อมูล และสร้างรายงานและแดชบอร์ดที่ถูกรวมเป็นพื้นที่ทำงานแอป 

เมื่อต้องการเรียนรู้เพิ่มเติม ดู[สิทธิ์การใช้งาน Power BI](service-admin-licensing-organization.md)

## <a name="tool-connectivity-preview"></a>เชื่อมต่อเครื่องมือ (ตัวอย่าง)

ภายใต้อยู่ภายใน enterprise พิสูจน์ Microsoft **กลไกจัดการ Vertipaq Analysis Services**สนับสนุนชุดข้อมูล Power BI Analysis Services มีโปรแกรม และแอปพลิเคชันไคลเอ็นต์และเครื่องมือสนับสนุนผ่านไลบรารีไคลเอ็นต์และ Api ที่สนับสนุนโพรโทคอล XMLA เปิดมาตรฐาน ชุดข้อมูล Power BI Premium รอง*แบบอ่านอย่างเดียว*การดำเนินการจาก Microsoft และแอปพลิเคชันไคลเอ็นต์ของบุคคลสาม และเครื่องมือ**จุดสิ้นสุด XMLA** 

เครื่องมือ Microsoft เช่น SQL Server Management Studio และตัวสร้างโพ รไฟล์ของ SQL Server และแอปของบริษัทเช่น DAX Studio และข้อมูลแสดงภาพแอปพลิเคชัน สามารถเชื่อมต่อกับ และสอบถามชุดข้อมูล Premium โดยใช้ XMLA, DAX, MDX, DMVs และติดตามเหตุการณ์
